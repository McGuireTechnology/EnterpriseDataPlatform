# Database Migrations

This repo uses Alembic for durable PostgreSQL DDL and keeps separate migration
histories for each EDP-owned database layer:

- `raw` targets `edp_raw` for source-system landing records and ingestion metadata. Raw data uses one schema per source system, such as `ad`, `m365`, or `fleettrack`.
- `ods` targets `edp_ods` for normalized current-state operational models.
- `vault` targets `edp_vault` for Data Vault hubs, links, satellites, and vault metadata.
- `mart` targets `edp_mart` for user-facing marts and refresh metadata.
- `app` targets `edp_app` for custom application state owned by EDP services.

The modeling layer lives in `postgres/models/`. Alembic imports the matching
SQLAlchemy `MetaData` object for each database so it can autogenerate migration
scripts from model changes.

Run migrations with:

```sh
make db-upgrade
```

For local development, copy `.env.example` to `.env` before starting Postgres so
Compose and Alembic use the same bootstrap user, password, and database URLs.

Create new revisions with:

```sh
alembic -c postgres/alembic.ini -n raw revision --autogenerate -m "describe raw change"
alembic -c postgres/alembic.ini -n ods revision --autogenerate -m "describe ods change"
alembic -c postgres/alembic.ini -n vault revision --autogenerate -m "describe data vault change"
alembic -c postgres/alembic.ini -n mart revision --autogenerate -m "describe data mart change"
alembic -c postgres/alembic.ini -n app revision --autogenerate -m "describe app change"
```

After editing models or migrations, check for model/database drift with:

```sh
make db-check
```

Always review autogenerated migrations before applying them. Alembic can compare
tables, columns, indexes, constraints, and types, but schema creation and some
PostgreSQL-specific operations may still need hand edits.

The local container also creates `airflow` and `superset` databases. Those are
component metadata databases and should be migrated by Airflow and Superset
themselves, not by EDP Alembic migrations.

Prefer dbt for transformation models, tests, and lineage once raw data and
stable relational structures exist. Prefer Alembic for database objects that
must be explicitly promoted, rolled back, or shared by services.
